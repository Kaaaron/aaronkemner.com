On Good Blueprints

From time to time, I am involved in discussions on whether it is possible to create “objectively good” Blueprints in Unreal Engine. The realistic answer is: Probably not. Not fully objectively at least. But that still doesn’t mean that I don’t have some strong opinions on how Blueprints should look, so here we are.

My philosophy of what makes Blueprints “good” is a result of about 8 years of using the visual programming language, initially just on my own and in small non-professional teams, but since 2020 also in a more professional setting. This means that while I obviously don’t know how you do your Blueprints I have seen a bunch of people (including myself of course) making very good and very bad decisions when creating these graphs.

Preamble

If you need something more in-depth than this blog post, I can recommend this Unreal Engine Style Guide. It was originally written for Unreal Engine 4 but it covers very fundamental topics so it still holds up in UE5. I do not agree with every single point in this style guide. The use of tooltips is redundant in most cases if your naming is descriptive enough, so making it mandatory does not make sense to me, and limiting the numbers of nodes per function is promoting unnecessarily splitting graphs up into single-use functions which increases complexity even more. But aside from a few examples like this, it is a very solid guide to use.

The Basics

On a very basic level, we want to avoid Blueprints from Hell, at the very least. To achieve this, we need to be thinking mainly about keeping our graphs readable.

A big part of achieving that is to thinkg of Blueprints as code. It is a node-based language and while that is inherently different to text-based code in some ways, that does not absolve us from trying to write good code.

Plug-Ins

I recommend at least using 2 Plug-Ins to aid with readability in Blueprints: FlatNodes and BlueprintAssist.

FlatNodes

FlatNodes is a plug-in that simplifies the display of editor nodes in Unreal. It doesn’t have any other implications besides the appearance, but this change alone already makes the graph less busy and more readable in comparison to the fancier appearance the nodes in Unreal take by default. And it is completely free!

BlueprintAssist

BlueprintAssist is a more functional help adding a bunch of new shortcuts and options to Unreals Node-Based editors. The one we care about most though is the formatting help Blueprint Assist provides at the click of a button.

Left-Handed vs. Helix

When talking about formatting, the only acceptable answer for me is left-handed formatting. Helixing is just worse.

But why is helixing worse? Why isn’t it?

If you are reading this, you are reading it from left to right first, then top to bottom, I assume. This is how you (generally) read any text using the latin alphabet. It has been forced into every one of our minds from the moment we learned to read, whether we want to or not. Because of this, Blueprints following the same flow are intuitively more understandable than helixing.

Helixing also forces the wires to overlap with the nodes themselves. In simple examples, this might still be fine, but as soon as you have more than a single nice chain it very quickly becomes hard to decipher which wire is going to which pin on which node.

And if you are saying “but I only have a simple case here, isn’t it fine then?”, this is just because you haven’t met your future self yet. It is not too unlikely that a lot of cases you deem simple now will be more complex later. Using left-handed formatting from the start keeps your graph readable even when new nodes need to get added to an oh-so-simple graph.

Wire Priority

“Align Wires, Not Nodes” is a excelet bit of the style guide metioned above and I agree with it whole-heartedly.

Execution Wires should have the highest priority. They should not cross, and they should always go from left to right.
